## Foundations (build first — shared infra)

### Compile-target reconciliation + dead-code purge (pbxproj source-of-truth) (~40h)
- **Unblocks:** 3 — Intel/RAG, 9 — Training/Scenarios, 10 — Security/OPSEC/Duress, 2 — LiDAR scan, 6 — Ops planning+AAR, 4 — Mesh-Meshtastic
- **Why:** The single most load-bearing finding across the census: the Xcode project membership is internally inconsistent. Entire trees (AI/ 30 files, Training/ 47, Scenarios/ 42, FieldOps/ 30, UI/ ~18, most of Security/ 35, all Interop/ 39, plus LiDARPipeline/HypothermiaCalc/GISOverlayProvider/IAPManager) are visible in the navigator but in ZERO Compile Sources phases, while the LIVE capture path (SpatialIntelligence/LiDAR) is NOT referenced at all. Duplicate type names (IntelCorpus/VerifyPipeline/EmbeddedVectorStore, 6x SearchBar, KeychainService x4, Scenario/ScenarioEvent/Decision/Skill) mean these files can't even coexist if added. Nothing else can be safely built until each target file's membership is deliberate: decide compile-vs-delete per tree, resolve symbol collisions, and harvest the few real files (AI/RAG corpora, VirtualVictims, ResourceSimulator) before deletion.
- **Key files:** ZeroDark.xcodeproj/project.pbxproj, Sources/MLXEdgeLLM/AI/RAG/IntelCorpus.swift, Sources/MLXEdgeLLM/Intelligence/IntelStubs.swift

### On-device embedding + persisted vector store (NLEmbedding + EmbeddedVectorStore.save/query + RRF) (~24h)
- **Unblocks:** 3 — Intel/RAG, 6 — Ops planning+AAR
- **Why:** The semantic half of RAG is non-functional everywhere: EmbeddedVectorStore.save() is empty, nothing queries the vectors, search() is BM25-only, and the 'real' AI/RAG embedding engine is actually a localhost HTTP client that can't run on iOS. A single canonical on-device embedding engine (Apple NLEmbedding) + a persisted, queryable vector store + cosine/RRF fusion unblocks photo-analysis ingestion retrieval, hybrid KB search, claim verification, SOP assistant, and few-shot selection.
- **Key files:** Sources/MLXEdgeLLM/Intelligence/IntelStubs.swift, Sources/MLXEdgeLLM/Intelligence/KnowledgeRAG.swift, Sources/MLXEdgeLLM/AI/RAG/EmbeddingEngine.swift, Sources/MLXEdgeLLM/AI/RAG/VectorStore.swift

### SwiftData/JSON persistence pass for ephemeral status stores (~12h)
- **Unblocks:** 8 — Coordination/Team-roster, 6 — Ops planning+AAR, 1 — Map+Breadcrumb
- **Why:** Several otherwise-real coordination/SAR features lose all state on restart: CheckInSystem (AAR pulls from it), SearchPatternManager sector coverage/assignments, LandSARSearchModel POA grid. For a multi-hour SAR op this volatility is a real gap. A shared persistence pattern (the module already uses SwiftData @Model and JSON elsewhere) applied to these stores hardens Coordination and the SAR pieces of Ops.
- **Key files:** Sources/MLXEdgeLLM/Coordination/CheckInSystem.swift, Sources/MLXEdgeLLM/Coordination/SearchPattern.swift, Sources/MLXEdgeLLM/Persistence/SwiftDataModels.swift

### Camera keyframe-ring retention for LiDAR colorization (~12h)
- **Unblocks:** 2 — LiDAR scan
- **Why:** The #1 cause of 'terrible renderings': every saved scan is a monochrome gray/green blob because export hardcodes UIColor.gray and drops normals, while a fully-working PointCloudColorizer is wired to nothing. To colorize at export time you must retain a small ring of ARFrames/keyframes during capture so color can be projected back. This single capture-side foundation unblocks USDZ/PLY/OBJ colorization across LiDARExportWriters, ReconWalkEngine, and PointCloudExporter.
- **Key files:** Sources/MLXEdgeLLM/SpatialIntelligence/LiDAR/LiDARCaptureEngine.swift, Sources/MLXEdgeLLM/SpatialIntelligence/LiDAR/LiDARExportWriters.swift, Sources/MLXEdgeLLM/LiDAR/Core/Colorization.swift

### Mesh key-distribution + persistent device identity keypair (~40h)
- **Unblocks:** 4 — Mesh-Meshtastic, 10 — Security/OPSEC/Duress, 5 — Mesh-Baofeng/VHF-UHF
- **Why:** Encryption is duplicated and broken across the comms/security blades: EncryptionManager.secureKeyDistribution seals keys under a throwaway ephemeral key no peer can derive AND regenerates identity each launch; ChannelManager's user-reachable 'rotate key' broadcasts an EMPTY Data() packet; LoRaComms/MeshCrypto generate random per-launch keys so peers can never decrypt. A persistent Keychain device identity + real ECDH peer-key exchange is the shared root for any honest encrypted mesh feature and for the duress/OPSEC keychain story.
- **Key files:** Sources/MLXEdgeLLM/CommunicationCore/EncryptionManager.swift, Sources/MLXEdgeLLM/CommunicationCore/ChannelManager.swift, Sources/MLXEdgeLLM/SecurityLayer/MeshCrypto/MeshCryptoManager.swift

### ZD.locationUpdate posting + BatteryProxy startSampling wiring (cross-cutting feed fixes) (~4h)
- **Unblocks:** 7 — Medical TCCC/MARCH, 1 — Map+Breadcrumb
- **Why:** Two tiny cross-cutting bugs silently kill multiple real features. (1) The 'ZD.locationUpdate' notification is observed by AltitudeTracker/HydrationCalc/SunCalculator/MoonPhase but is NEVER posted, so AltitudeTracker runs permanently on altitude=0 (its whole AMS UI is dead) and hydration altitude term never engages. (2) BatteryProxy.startSampling() only runs inside an orphaned view, so MapTabView's battery trend/minutes-remaining never updates past defaults — a 1-line app-launch call fixes a live readout. Both are wire-real-engines-into-the-app fixes.
- **Key files:** Sources/MLXEdgeLLM/Services/LocationManager.swift, Sources/MLXEdgeLLM/Services/AltitudeTracker.swift, Sources/MLXEdgeLLM/Logistics/BatteryProxy.swift

## LiDAR rendering fix (flagship)

Capture is real, but the DISPLAY/EXPORT path throws away the good data at two persistence bottlenecks (points.bin = positions-only 12B/pt; scan.usdz = flat gray) plus three render-time material overrides. Verified all line numbers and APIs. The fastest, highest-impact wins are pure wiring: (1) shade the USDZ mesh with ARMeshGeometry.normals (already on the anchor, currently dropped at LiDARExportWriters L82-132) so the model reads as a lit surface instead of a flat gray blob; (2) re-render the Terrain3d point cloud with normals + a real lighting model instead of unlit cyan dots. Color is a second tier: PointCloudColorizer (Colorization.swift) is fully written but has ZERO call sites (confirmed by grep) — wiring it requires retaining camera keyframes during scan (ARFrames are NOT currently kept, LiDARCaptureEngine L1852) and a sidecar color file, because points.bin's positions-only format is read by 6+ consumers and must not break. CRITICAL coupling: ScanGalleryView's TacticalSceneView (L370-382, L429-446) force-overwrites every geometry's material with fixed tactical green, so any USDZ color/normal work is invisible in that viewer until the override becomes conditional. Live-scan tweaks (environmentTexturing, debugOptions) are low-effort polish. estimateNormals (PointCloudEngine, called L715) already produces per-point normals that are currently discarded after plane detection — reuse them for both the sidecar and analysis. Fix order below front-loads the wiring-only wins and sequences the format/retention changes so no existing reader breaks.
